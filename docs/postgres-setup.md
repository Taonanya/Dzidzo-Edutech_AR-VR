# PostgreSQL Setup and Database Guide

## Purpose

This repository currently contains incomplete Supabase-based authentication code, but the actual application structure is a static frontend with sign-in and sign-up forms. The practical database migration path is:

1. Stop treating the browser as the database client.
2. Use PostgreSQL as the application database.
3. Move authentication and user queries to a backend service or API layer.
4. Store password hashes, never raw passwords.

This guide documents how to install PostgreSQL on Windows, create the database, run the schema scripts in this repository, and prepare the project for a proper Postgres-backed implementation.

## What Is Changing

The current repo has these database-related problems:

- `supabaseClient.js` expects `process.env` in a browser context.
- `auth.js` writes and reads users directly from the client.
- `auth.js` compares plaintext passwords.
- `signin.js` and `signup.js` look like server handlers, but this repo does not contain the backend structure needed to run them.

The replacement direction is:

- PostgreSQL becomes the source of truth.
- A backend connects to PostgreSQL using a server-side connection string.
- The frontend sends sign-in and sign-up requests to that backend.
- Passwords are hashed before storage.

## Download And Install PostgreSQL On Windows

### Option 1: Official Windows Installer

1. Go to the PostgreSQL downloads page:
   `https://www.postgresql.org/download/windows/`
2. Follow the link to the EnterpriseDB installer.
3. Download the latest stable PostgreSQL installer for Windows.
4. Run the installer.

Recommended installation choices:

- Install `PostgreSQL Server`
- Install `pgAdmin`
- Install `Command Line Tools`
- Keep the default port: `5432`
- Set a strong password for the `postgres` superuser

During setup, note these values:

- host: `localhost`
- port: `5432`
- superuser: `postgres`
- password: the one you choose during install

### Option 2: Package Manager

If you prefer package managers, you can use one of these:

```powershell
winget install PostgreSQL.PostgreSQL
```

or

```powershell
choco install postgresql
```

If you use a package manager, still verify that:

- PostgreSQL is running as a Windows service
- `psql` is on your `PATH`
- port `5432` is open locally

## Verify Installation

Open PowerShell and run:

```powershell
psql --version
```

You should see a PostgreSQL version string.

If that fails, open the PostgreSQL shell from the Start Menu or add PostgreSQL's `bin` directory to your system `PATH`.

## Create The Database And App User

Connect as the `postgres` superuser:

```powershell
psql -U postgres -h localhost
```

Then run:

```sql
CREATE ROLE dzidzo_app WITH LOGIN PASSWORD 'change_this_password';
CREATE DATABASE dzidzo OWNER dzidzo_app;
\c dzidzo
GRANT ALL PRIVILEGES ON DATABASE dzidzo TO dzidzo_app;
```

If you want a stricter setup, create the database as `postgres` and grant only the permissions your app needs later.

## Connection String Format

Use this format in your backend environment variables:

```env
DATABASE_URL=postgresql://dzidzo_app:change_this_password@localhost:5432/dzidzo
```

Common parts:

- username: `dzidzo_app`
- password: `change_this_password`
- host: `localhost`
- port: `5432`
- database: `dzidzo`

## Run The Database Scripts In This Repo

From the repository root:

```powershell
psql -U dzidzo_app -h localhost -d dzidzo -f database/postgres/01_schema.sql
psql -U dzidzo_app -h localhost -d dzidzo -f database/postgres/03_indexes.sql
psql -U dzidzo_app -h localhost -d dzidzo -f database/postgres/02_seed.sql
```

Recommended order:

1. schema
2. indexes
3. seed data

## What The SQL Files Do

### `01_schema.sql`

### `02_seed.sql`

Inserts example users for:

- admin

- student

Important:

- the seed file uses placeholder password hashes
- replace them with hashes generated by your backend before production use

### `03_indexes.sql`

Adds:

- an index for user role queries
- an index for created date sorting

The `email` uniqueness is already enforced in the table definition.

## Suggested Backend Connection Pattern

Do not connect to PostgreSQL directly from static browser JavaScript.

Use this structure instead:

1. frontend form submits to `/api/auth/signup`
2. backend validates input
3. backend hashes password
4. backend inserts into PostgreSQL
5. frontend form submits to `/api/auth/signin`
6. backend fetches user by email
7. backend verifies password hash
8. backend returns a session or token

## Example Environment Variables

Add these to a backend `.env` file when you build the server layer:

```env
DATABASE_URL=postgresql://dzidzo_app:change_this_password@localhost:5432/dzidzo
DB_HOST=localhost
DB_PORT=5432
DB_NAME=dzidzo
DB_USER=dzidzo_app
DB_PASSWORD=change_this_password
APP_ENV=development
```

## Notes On Password Storage

Never store raw passwords in PostgreSQL.

Store only password hashes produced by a server-side password hashing algorithm such as:

- bcrypt
- argon2

For this project, `bcrypt` is acceptable as a starting point if you are building a Node backend.
